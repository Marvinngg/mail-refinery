# 共识文档

每次开始工作前必须读这个文档，确保理解一致。

---

## 一、系统定位

企业邮件 AI 精炼管线。把 IMAP 邮箱数据精炼为按 文件夹→项目→话题 组织的 Markdown 知识库，供 AI Agent 消费。

## 二、三层架构

### Layer 1: 数据获取（sync.js）
- IMAP 按文件夹增量拉取邮件
- 正文提取（findPartByType 处理 multipart）
- 回复体分割（split-replies.js，识别引用格式拆成对话轮次）
- 附件下载到本地（data/attachments/{folder}/{uid}/文件名）
- 写入 SQLite（emails + email_turns 表）

### Layer 2: 数据处理（process.js）
从数据库读取，调本地模型，产出 Markdown 文件。
步骤详见第四节。

### Layer 3: 应用层
未实现。最终 Claude 通过读 md 文件 + 查 SQLite 消费数据。

### 入口
- run.js：循环服务，每 5 分钟一轮（sync → 判断全量/增量 → process）
- 自动判断：events/ 有内容+state 有效→增量，否则→全量

## 三、数据产出格式

```
data/events/
├── {文件夹}/
│   ├── _index.md                    ← 文件夹级概览
│   ├── {项目名}/
│   │   ├── _index.md                ← 项目级概览
│   │   ├── {话题名}/event.md        ← 事件脉络文档
│   │   └── ...
│   └── ...
└── _process_state.json              ← 处理状态
```

event.md 是最终产出，必须包含：
- 概要、参与者、时间线（倒序）、附件清单
- 附件的关键内容摘要
- 附件的本地路径（供定位）

## 四、process.js 处理步骤

按文件夹独立处理，步骤间有依赖关系：

### 步骤 1：项目+话题识别
- 输入：文件夹内所有技术线程的元信息 + 内容提示（turn_index=0 前 200 字）
- 过滤：广告/垃圾邮件按规则排除
- 分批：每批 12 条线程
- 模型输出：JSON（项目→话题两层结构）
- **JSON 解析失败时循环重试直到成功**

### 步骤 2：event.md 生成
- 每个话题独立生成
- 输入：话题下所有邮件的 turn_index=0（回复链中间） 或 所有 turn（链首/转发）+ 附件提取内容
- 分割不充分时标注给模型兜底
- 输入长度限制 20000 字符，超出截断最早的邮件
- 模型输出：Markdown 文本，直接写文件

### 步骤 3：项目级 _index.md
- 代码生成，不调模型
- 从 event.md 提取概要

### 步骤 4：项目合并
- 目的：把步骤 1 分批产生的相似项目合并
- 输入：每个项目的 _index.md 内容（前 100 行）
- 分批：每批 30 个项目
- 多轮：循环直到 ≤30 个，再做最终合并
- 模型输出：JSON（合并方案）
- **JSON 解析失败时循环重试**
- 合并后文件系统重组：按合并结果的 topics 列表从旧目录复制 event.md 到新目录，删除旧目录

### 步骤 5：文件夹级 _index.md
- 代码生成

### 步骤 6：跨文件夹关联
- 模型判断

### 步骤 7：保存 state
- 必须执行，不管前面有没有失败

## 六、已知问题

### 1. 步骤 4 合并后话题丢失
- 现象：288 项目 848 话题 → 合并后 1 项目 75 话题
- 原因：多轮合并中模型返回的 original_indices 可能没覆盖全部项目，导致部分项目的 topics 没被传递到下一轮
- 解法：合并后校验 indices 覆盖率，遗漏的补回来
- 状态：未修

### 2. 模型输入输出无日志
- 现象：无法排查模型返回了什么
- 解法：在 lib/model.js 的 callModel 里加日志，记录每次调用的输入摘要和完整输出
- 日志要求：有效信息，不输出噪音（如 PDF 提取报错不需要每次都打）
- 状态：未修

### 3. event.md 时间线未显式标注邮件 UID
- 现象：附件路径里包含 UID（如 attachments/INBOX/9593/文件名），可以反查数据库。但时间线正文里没有显式标注 UID
- 影响：反查需要从路径解析，不够直观
- 解法：在时间线条目里显式标注 UID
- 状态：未修

### 4. 附件路径以 UID 命名
- 现象：data/attachments/INBOX/10004/文件名，UID 对用户不可读
- 影响：如果 event.md 里有完整路径可以找到，但依赖路径而不是语义关联
- 状态：可接受，不急

### 5. PDF 提取失败
- 现象：pdf-parse 模块在 Node.js 22 下报 TypeError
- 影响：PDF 附件内容无法提取，event.md 中缺少 PDF 附件的内容摘要
- 状态：未修

### 6. download.js 和 sync.js 附件下载职责重叠
- 现象：sync.js 里有附件下载逻辑，download.js 也有
- 影响：维护成本高，容易不一致
- 状态：未处理

### 7. 增量追加的 event.md 条目渲染污染（严重）
- 现象：`appendToEventMd` 追加进 event.md 的新条目里，`\n` 是字面字符串不是换行，且每块末尾常挂一个孤立的 `}`。示例（`263Global - Peace 100G SG-MRS BAFO/event.md`）：
  ```
  ### 2026-04-23 — Darwin Ho [收件]\n感谢支持，附上订单表单…
  }
  ```
- 影响范围：所有被增量追加过的话题。已确认受污染：
  - `INBOX/项目「商务询价与报价」/[商务询价与报价]/event.md`（多处，含大段引用链污染）
  - `INBOX/项目「商务询价与报价」/263Global - Peace 100G SG-MRS BAFO/event.md`（开头 3 条新增里 2 条坏）
  - 其他被"追加 N 封"命中的话题（见 run.log）
- 根因：模型实际返回 JSON（`{"updated_summary": "...", "new_entries": "...\n..."}`），但 JSON 里 new_entries 字段经常塞进整封引用链邮件，裸双引号/控制字符导致 `parseJSON` 崩。此时走 markdown fallback，正则 `/(###[\s\S]*)/` 直接从**原始 JSON 字符串**抓内容，没反转义 `\n`，也没剥尾部 `"}`。
- 候选解法：
  1. 最小：fallback 里识别疑似 JSON 串，反转义 `\n` / `\"` / `\\` 并剥 `"}`。改 ~10 行，只救渲染，不治本。
  2. 较稳：改 prompt 让模型**只输出 markdown**，不用 JSON。summary 用正则提取。
  3. 治本：追加前先把单封邮件正文截到 500 字以内，buildEmailEntry 要切干净引用链，JSON 字段体量压下去。
- 建议：选方案 2。9B 模型对长自由文本出 JSON 天然脆弱。
- 附带：已污染文件需要重跑那几个话题的 event.md 全量生成（不是整库全量），否则 AI Agent 读到的是坏数据。
- 状态：未修，等用户决策方案

## 六、工作纪律

1. 不做没跟用户确认的改动
2. 不删除 data/ 下的任何数据（PreToolUse hook 拦截 rm -r）
3. 改代码前说清楚改了什么、为什么改、影响哪个模块
4. 跑测试前说清楚测的是哪个模块、期望结果、怎么验证
5. 跑完后如实汇报，不传递乐观信息
6. 部署运行后是监督者，不手动绕过系统执行
